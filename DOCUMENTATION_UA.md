# Документація до проєкту Alarm Clock

Цей документ написаний як підготовка до усного захисту. Його ціль - не тільки описати, що робить проєкт, а й пояснити, як працює код, навіщо потрібен кожен файл і що відповісти, якщо запитають про конкретну функцію.

## 1. Короткий опис проєкту

Проєкт - це будильник на Arduino UNO. Він використовує LCD дисплей 16x2 для показу інформації, RTC модуль DS1302 для роботи з датою і часом, DHT11 для температури і вологості, кнопки для керування, buzzer для звукового сигналу і LED для індикації активного будильника.

Основні можливості:

- показ поточного часу;
- показ поточної дати;
- показ температури і вологості;
- налаштування часу кнопками;
- налаштування дати кнопками;
- налаштування будильника;
- збереження налаштувань будильника в EEPROM;
- звуковий сигнал будильника через buzzer;
- LED індикатор активного будильника;
- snooze, тобто відкладання будильника на 5 хвилин;
- секундомір;
- countdown timer, тобто таймер зворотного відліку;
- factory reset для скидання налаштувань.

Проєкт зроблений як state machine. Це означає, що програма має набір станів або екранів, і в кожен момент часу активний тільки один з них: годинник, дата, датчики, налаштування часу, налаштування будильника тощо.

## 3. Підключення

Всі піни задані у файлі `include/config.h`. Якщо на захисті питають, де налаштовуються піни, відповідь: "Усі апаратні піни винесені в `config.h`, щоб не шукати їх по всьому коду".

| Пристрій | Пін пристрою | Пін Arduino |
|---|---:|---:|
| LCD I2C | SDA | A4 |
| LCD I2C | SCL | A5 |
| LCD I2C | VCC | 5V |
| LCD I2C | GND | GND |
| DS1302 RTC | DAT | D7 |
| DS1302 RTC | CLK | D6 |
| DS1302 RTC | RST | D8 |
| DS1302 RTC | VCC | 5V |
| DS1302 RTC | GND | GND |
| DHT11 | DATA / S | D5 |
| DHT11 | VCC / + | 5V |
| DHT11 | GND / - | GND |
| MODE button | signal | D2 |
| SELECT button | signal | D3 |
| UP button | signal | D4 |
| DOWN button | signal | D9 |
| Buzzer | signal / + | D10 |
| Buzzer | GND / - | GND |
| LED | через резистор | D11 |

Кнопки підключені через `INPUT_PULLUP`. Це означає:

- кнопка не натиснута - Arduino читає `HIGH`;
- кнопка натиснута - Arduino читає `LOW`.

Це важливо, бо в коді натискання визначається саме переходом з `HIGH` у `LOW`.

## 5. Загальна структура проєкту

```txt
include/
  config.h
  context.h
  lcd_wrapper.h
  rtc_wrapper.h
  screens.h
  sensors.h

  main.cpp
  lcd_wrapper.cpp

```

Головна ідея структури:

- `main.cpp` тільки запускає state machine;
- `context.h` містить усі змінні стану програми;
- `screens.h` описує всі екрани;
- `src/screens/init.cpp` виконує ініціалізацію;
- `src/screens/app.cpp` містить логіку екранів, кнопок, будильника, таймера і секундоміра;
- `lcd_wrapper.cpp` ізолює роботу з LCD;
- `rtc_wrapper.cpp` ізолює роботу з часом, RTC і EEPROM snapshot;
- `sensors.cpp` ізолює роботу з DHT11.

## 6. Керування кнопками

| Кнопка | Основна дія |
|---|---|
| MODE | Перехід на наступний екран |
| SELECT | Вхід у редагування, підтвердження, старт/пауза |
| UP | Збільшити значення або reset секундоміра |
| DOWN | Зменшити значення |

Поведінка залежить від активного екрана.

На екрані будильника:

- `SELECT` відкладає сигнал на 5 хвилин;
- `MODE`, `UP` або `DOWN` вимикають сигнал.

На екрані секундоміра:

- `SELECT` старт або пауза;
- `UP` скидає секундомір.

На екрані таймера:

- `UP` і `DOWN` змінюють хвилини або секунди;
- `SELECT` переходить від хвилин до секунд, потім стартує таймер;
- під час роботи `SELECT` ставить на паузу або продовжує.

Цей файл містить усі налаштування пінів і базових параметрів.

```cpp
#define BAUD_RATE 9600
```

```cpp
#define RTC_DAT_PIN 7
```

DS1302 не працює через I2C. Він має окремі лінії `DAT`, `CLK`, `RST`.

### `BUZZER_PIN`

```cpp
#define BUZZER_PIN 10
```

Пін для buzzer. Через нього функція `tone()` генерує звук.

### `DHT_PIN`

```cpp
#define DHT_PIN 5
```

Пін даних для DHT11.

### `BTN1_PIN`, `BTN2_PIN`, `BTN3_PIN`, `BTN4_PIN`

```cpp
#define BTN1_PIN 2
#define BTN2_PIN 3
#define BTN3_PIN 4
#define BTN4_PIN 9
```

У коді вони перейменовуються в:

- `BTN_MODE`;
- `BTN_SELECT`;
- `BTN_UP`;
- `BTN_DOWN`.

### `STATUS_LED_PIN`

```cpp
#define STATUS_LED_PIN 11
```

Пін LED. LED світиться, коли будильник увімкнений, і блимає під час сигналу.

### `LCD_I2C_ADDRESS`, `LCD_ROWS`, `LCD_COLS`

## 9. Файл `include/context.h`

Це один із найважливіших файлів. У ньому описані структури, які зберігають стан програми.

### `struct button_state`

```cpp
struct button_state {
    byte pin;
    bool last_state;
    unsigned long last_change;
};
```

Ця структура потрібна для кнопок.

Поля:

- `pin` - номер піна Arduino;
- `last_state` - попередній стан кнопки, `HIGH` або `LOW`;
- `last_change` - час останньої зміни стану, використовується для debounce.

Debounce потрібен тому, що механічна кнопка при натисканні може кілька разів швидко змінити сигнал. Без debounce одне натискання могло б спрацювати як багато натискань.

### `struct context`

`context` - це головна структура стану програми. Вона передається у всі screen-функції.

Основна ідея: замість великої кількості глобальних змінних усе зберігається в одному об'єкті.

Групи полів:

#### Поточний екран

```cpp
byte current_screen;
```

Зберігає, який екран зараз активний.

#### Кнопки

```cpp
button_state mode_button;
button_state select_button;
button_state up_button;
button_state down_button;
```

Стан кожної з чотирьох кнопок.

#### Дані датчика

```cpp
float temperature;
int humidity;
unsigned long last_sensor_read;
```

Зберігають останню температуру, вологість і час останнього читання DHT11.

#### Налаштування часу

```cpp
bool setting_time;
byte time_field;
byte set_hour;
byte set_minute;
```

- `setting_time` показує, чи користувач зараз редагує час;
- `time_field` показує, яке поле редагується: години або хвилини;
- `set_hour`, `set_minute` - тимчасові значення до збереження.

#### Налаштування дати

```cpp
bool setting_date;
byte date_field;
byte set_day;
byte set_month;
int set_year;
```

Аналогічно до часу, тільки для дня, місяця і року.

#### Будильник

```cpp
bool setting_alarm;
byte alarm_field;
byte alarm_hour;
byte alarm_minute;
bool alarm_enabled;
bool alarm_ringing;
int last_alarm_key;
```

- `alarm_hour`, `alarm_minute` - час будильника;
- `alarm_enabled` - чи увімкнений будильник;
- `alarm_ringing` - чи зараз звучить сигнал;
- `last_alarm_key` потрібен, щоб будильник не запускався багато разів протягом однієї хвилини.

#### Snooze

```cpp
bool snooze_active;
byte snooze_hour;
byte snooze_minute;
```

Зберігає час, на який відкладено будильник.

#### Stopwatch

```cpp
bool stopwatch_running;
unsigned long stopwatch_started_at;
unsigned long stopwatch_elapsed_before_start;
```

Потрібно для секундоміра.

- `stopwatch_running` - чи секундомір зараз працює;
- `stopwatch_started_at` - момент старту;
- `stopwatch_elapsed_before_start` - накопичений час перед останнім стартом.

#### Timer

```cpp
bool timer_setting;
byte timer_field;
byte timer_minutes;
byte timer_seconds;
bool timer_running;
bool timer_done;
unsigned long timer_ends_at;
unsigned long timer_remaining_ms;
```

Поля для countdown timer:

- режим налаштування;
- хвилини і секунди;
- чи працює таймер;
- чи завершився таймер;
- коли таймер має завершитися;
- скільки часу залишилось.

## 10. Файл `include/screens.h`

У цьому файлі описано enum екранів і прототипи функцій.

### `enum screen`

Це список усіх станів програми. Кожен стан відповідає одному екрану або режиму.

Якщо питають, чому використовується enum: enum робить код читабельним. Замість чисел `0`, `1`, `2` використовуються назви `CLOCK_SCR`, `SET_ALARM_SCR` тощо.

### Прототипи функцій екранів

Наприклад:

```cpp
enum screen clock_screen(struct context *ctx);
enum screen set_alarm_screen(struct context *ctx);
```

Кожна така функція:

1. отримує `context`;
2. виконує логіку свого екрана;
3. повертає наступний активний екран.

## 11. Файл `src/main.cpp`

Цей файл спеціально зроблений коротким.

### `int main()`

```cpp
int main() {
    struct context context;
    enum screen screen = INIT_SCR;

    for (;;) {
        switch (screen) {
            ...
        }
    }
}
```

Що відбувається:

1. Створюється структура `context`, де буде зберігатися стан програми.
2. Початковий екран - `INIT_SCR`.
3. Запускається нескінченний цикл `for (;;)`.
4. Через `switch` викликається функція активного екрана.
5. Кожна функція повертає наступний екран.

Це класична реалізація state machine.

Як пояснити на захисті:

"`main.cpp` не містить бізнес-логіки. Він тільки перемикає стани. Вся логіка винесена в screen-функції та wrapper-файли".

## 12. Файл `src/screens/init.cpp`

Цей файл відповідає за початкову ініціалізацію.

### `context_init(struct context *ctx)`

Ця функція заповнює `context` початковими значеннями.

Що вона робить:

- встановлює початковий екран `CLOCK_SCR`;
- налаштовує button state для чотирьох кнопок;
- обнуляє температуру і вологість;
- вимикає режими редагування часу, дати і будильника;
- задає дефолтний будильник `07:00 OFF`;
- вимикає snooze;
- скидає секундомір;
- задає таймер на `01:00`.

Чому це важливо:

Якщо не ініціалізувати всі поля, у пам'яті можуть бути випадкові значення. Це може призвести до дивної поведінки меню.

### `init_screen(struct context *ctx)`

Це перший screen, який викликається з `main.cpp`.

Основні дії:

```cpp
init();
context_init(ctx);
Wire.begin();
Serial.begin(BAUD_RATE);
```

`init()` - Arduino-ініціалізація, потрібна, бо використовується власний `main()`, а не стандартні `setup()` і `loop()`.

Далі налаштовуються піни:

```cpp
pinMode(BTN_MODE, INPUT_PULLUP);
pinMode(BTN_SELECT, INPUT_PULLUP);
pinMode(BTN_UP, INPUT_PULLUP);
pinMode(BTN_DOWN, INPUT_PULLUP);
pinMode(BUZZER_PIN, OUTPUT);
pinMode(STATUS_LED_PIN, OUTPUT);
```

Потім ініціалізуються модулі:

```cpp
lcd_init();
lcd_backlight(true);
clock_init();
sensors_init();
load_alarm_settings(ctx);
```

Наприкінці показується стартове повідомлення:

```txt
Alarm Clock
Ready
```

Після цього функція повертає `CLOCK_SCR`, і програма переходить на екран годинника.

## 13. Файл `src/lcd_wrapper.cpp`

Цей файл приховує роботу з бібліотекою `LiquidCrystal_I2C`.

Глобально всередині файлу створюється LCD-об'єкт:

```cpp
static LiquidCrystal_I2C lcd(LCD_I2C_ADDRESS, LCD_COLS, LCD_ROWS);
```

Ключове слово `static` означає, що цей об'єкт доступний тільки всередині `lcd_wrapper.cpp`.

### `lcd_init()`

```cpp
void lcd_init() {
    lcd.init();
}
```

Ініціалізує LCD дисплей.

### `lcd_clear()`

Очищає весь дисплей.

### `lcd_set_cursor(int y, int x)`

```cpp
lcd.setCursor(x, y);
```

Встановлює курсор у позицію.

У wrapper параметри передаються як `y, x`, тобто спочатку рядок, потім колонка.

### `lcd_print(const char* text)`

Виводить текст у поточну позицію курсора.

### `lcd_print_at(int y, int x, const char* text)`

Комбінована функція:

1. ставить курсор;
2. друкує текст.

### `lcd_print_line(int y, const char* text)`

Одна з найкорисніших функцій.

Вона друкує рядок шириною 16 символів:

```cpp
snprintf(line, sizeof(line), "%-16s", text);
```

`%-16s` означає: надрукувати текст і доповнити пробілами до 16 символів.

Навіщо це потрібно:

LCD не очищає старі символи автоматично. Якщо спочатку було `Temperature`, а потім друкувати `Time`, то частина старого тексту може залишитися. `lcd_print_line()` прибирає цю проблему.

### `lcd_clear_line(int y)`

Очищає один рядок дисплея, друкуючи порожній рядок через `lcd_print_line()`.

### `lcd_backlight(bool state)`

Вмикає або вимикає підсвітку LCD.

Створює об'єкт датчика DHT11.

Читає вологість.

```cpp
if (!isnan(new_temperature) && new_humidity >= 0)
```

Тобто нові значення записуються в `context` тільки якщо датчик повернув нормальні дані.

## 15. Файл `src/rtc_wrapper.cpp`

Цей файл відповідає за час, дату, RTC DS1302 і EEPROM snapshot часу.

### Важлива ідея

У проєкті використовується DS1302 RTC, але для стабільного відновлення останнього встановленого часу також використовується EEPROM snapshot.

Коли користувач зберігає час або дату:

1. значення записується в програмний годинник;
2. значення записується в EEPROM;
3. значення записується в DS1302.

Коли Arduino стартує:

1. спочатку пробує прочитати snapshot з EEPROM;
2. якщо snapshot є і він валідний, стартує з нього;
3. якщо snapshot немає, пробує прочитати RTC;
4. якщо і RTC невалідний, ставить дефолтний час.

Це зроблено, щоб після перезапуску Arduino відновлювала останній встановлений час.

### EEPROM адреси

```cpp
#define EEPROM_TIME_MAGIC_ADDR 10
#define EEPROM_TIME_YEAR_ADDR 11
#define EEPROM_TIME_MONTH_ADDR 12
#define EEPROM_TIME_DAY_ADDR 13
#define EEPROM_TIME_HOUR_ADDR 14
#define EEPROM_TIME_MINUTE_ADDR 15
#define EEPROM_TIME_SECOND_ADDR 16
#define EEPROM_TIME_MAGIC_VALUE 0x5C
```

EEPROM зберігає:

- рік;
- місяць;
- день;
- години;
- хвилини;
- секунди;
- magic byte.

Magic byte потрібен, щоб зрозуміти, чи в EEPROM вже є валідні дані.

### `static ThreeWire rtc_wire(...)`

DS1302 працює через ThreeWire:

```cpp
static ThreeWire rtc_wire(RTC_DAT_PIN, RTC_CLK_PIN, RTC_RST_PIN);
```

Порядок параметрів:

```txt
DAT, CLK, RST
```

### `static RtcDS1302<ThreeWire> rtc(rtc_wire)`

Створює об'єкт RTC.

### `datetime_is_valid(const RtcDateTime &date_time)`

Перевіряє, чи дата і час нормальні.

Перевіряється:

- рік від 2024 до 2099;
- місяць від 1 до 12;
- день від 1 до 31;
- години від 0 до 23;
- хвилини від 0 до 59;
- секунди від 0 до 59;
- внутрішня перевірка `date_time.IsValid()`.

Навіщо це потрібно:

RTC іноді може повернути невалідні значення, якщо він не був налаштований або є проблема з живленням. Ця функція не дає таким значенням зламати логіку.

### `fallback_datetime()`

Повертає дефолтну дату і час:

```txt
22.05.2026 00:00:00
```

Це запасний варіант, якщо немає валідних даних ні з EEPROM, ні з RTC.

### `set_app_clock(const RtcDateTime &date_time)`

Запускає внутрішній програмний годинник.

Зберігає:

- базовий час `app_clock_base`;
- момент, коли цей час був встановлений `app_clock_set_at`;
- прапорець `app_clock_valid`.

Після цього функція `safe_now()` може рахувати поточний час через `millis()`.

### `save_clock_snapshot(const RtcDateTime &date_time)`

Записує час і дату в EEPROM.

Використовується `EEPROM.update()`, а не `EEPROM.write()`.

Різниця:

- `write()` завжди записує байт;
- `update()` записує тільки якщо значення змінилося.

Це краще для EEPROM, бо EEPROM має обмежену кількість циклів запису.

### `load_clock_snapshot(RtcDateTime &date_time)`

Пробує прочитати дату і час з EEPROM.

Алгоритм:

1. Перевіряє magic byte.
2. Якщо magic byte неправильний, повертає `false`.
3. Читає рік, місяць, день, години, хвилини, секунди.
4. Створює `RtcDateTime`.
5. Перевіряє валідність.
6. Якщо все добре, записує результат у параметр `date_time` і повертає `true`.

### `safe_now()`

Це центральна функція отримання часу.

Алгоритм:

1. Якщо програмний годинник вже валідний, повертає:

```cpp
app_clock_base + elapsed_seconds
```

2. Якщо програмний годинник ще не встановлений, читає RTC.
3. Якщо RTC валідний, використовує RTC.
4. Якщо RTC невалідний, використовує fallback.

Це функція для безпечного отримання часу без ризику отримати сміттєві значення.

### `to_dt(const RtcDateTime &date_time)`

Перетворює тип `RtcDateTime` з бібліотеки RTC у власну структуру `dt`.

Навіщо:

У всій програмі зручніше використовувати просту структуру:

```cpp
struct dt {
    byte day;
    byte month;
    int year;
    byte hours;
    byte minutes;
    byte seconds;
};
```

### `clock_init()`

Ініціалізує RTC і стартовий час.

Алгоритм:

1. `rtc.Begin()` запускає RTC.
2. Вимикає write protection:

```cpp
rtc.SetIsWriteProtected(false);
```

3. Запускає RTC:

```cpp
rtc.SetIsRunning(true);
```

4. Пробує завантажити час з EEPROM.
5. Якщо EEPROM валідний, використовує його.
6. Якщо ні, пробує RTC.
7. Якщо RTC теж невалідний, ставить fallback.

### `set_date(...)`

Змінює тільки дату, але залишає поточний час.

Вона викликає `now()`, бере поточні години, хвилини і секунди, і потім викликає `set_datetime()`.

### `set_time(...)`

Змінює тільки час, але залишає поточну дату.

### `set_datetime(...)`

Головна функція запису дати і часу.

Вона:

1. створює `RtcDateTime`;
2. оновлює програмний годинник;
3. записує snapshot в EEPROM;
4. записує дату і час у RTC;
5. запускає RTC.

Це основна функція, яку викликають `Set Time` і `Set Date`.

### `get_day()`, `get_month()`, `get_year()`

Повертають окремі частини дати.

### `get_hours()`, `get_minutes()`, `get_seconds()`

Повертають окремі частини часу.

### `now()`

Повертає весь поточний час і дату як `struct dt`.

Цю функцію використовують екрани, будильник, snooze, дата і час.

### `reset_saved_datetime()`

Скидає magic byte в EEPROM для збереженого часу.

Використовується у Factory Reset.

Після цього при наступному старті програма не буде використовувати старий EEPROM snapshot.

## 16. Файл `src/screens/app.cpp`

Це найбільший файл, у ньому знаходиться основна логіка програми.

### EEPROM для будильника

```cpp
#define EEPROM_MAGIC_ADDR 0
#define EEPROM_ALARM_HOUR_ADDR 1
#define EEPROM_ALARM_MINUTE_ADDR 2
#define EEPROM_ALARM_ENABLED_ADDR 3
#define EEPROM_MAGIC_VALUE 0xA6
```

У EEPROM зберігаються:

- година будильника;
- хвилина будильника;
- чи будильник увімкнений;
- magic byte для перевірки валідності.

### `days_in_month(byte month, int year)`

Повертає кількість днів у місяці.

Враховує:

- лютий;
- високосний рік;
- місяці з 30 днями;
- місяці з 31 днем.

Використовується при налаштуванні дати, щоб не можна було вибрати, наприклад, 31 квітня.

### `read_pressed(struct button_state *button)`

Обробляє натискання кнопки з debounce.

Алгоритм:

1. Читає поточний стан піна.
2. Якщо стан змінився і пройшло більше 60 мс, зміна приймається.
3. Якщо новий стан `LOW`, це означає натискання.
4. Функція повертає `true` тільки один раз на натискання.

Це важливо, бо кнопки підключені через `INPUT_PULLUP`.

### `save_alarm_settings(struct context *ctx)`

Записує налаштування будильника в EEPROM.

Записує:

- magic byte;
- годину;
- хвилину;
- `alarm_enabled`.

### `load_alarm_settings(struct context *ctx)`

Завантажує налаштування будильника з EEPROM.

Якщо magic byte неправильний, ставить стандартні значення:

```txt
07:00 OFF
```

Також перевіряє, щоб:

- година не була більше 23;
- хвилина не була більше 59.

### `next_screen(struct context *ctx)`

Перемикає програму на наступний екран.

Якщо поточний екран - останній або екран сигналу, повертає на `CLOCK_SCR`.

Також вимикає режими редагування:

```cpp
ctx->setting_time = false;
ctx->setting_date = false;
ctx->setting_alarm = false;
```

І очищає LCD.

### `read_sensor(struct context *ctx)`

Читає DHT11, але не частіше ніж раз на 2 секунди.

Чому:

DHT11 не можна читати дуже часто. Якщо читати занадто часто, він може повертати помилки.

Функція:

1. перевіряє, чи пройшло 2000 мс;
2. читає температуру і вологість;
3. якщо дані валідні, записує їх у `context`.

### `start_set_time(struct context *ctx)`

Починає редагування часу.

Бере поточний час з `now()` і копіює його в:

- `ctx->set_hour`;
- `ctx->set_minute`.

Після цього користувач змінює не реальний час одразу, а тимчасові значення.

### `save_set_time(struct context *ctx)`

Зберігає налаштований час.

Алгоритм:

1. Бере поточну дату через `now()`.
2. Викликає `set_datetime()`, передаючи стару дату і новий час.
3. Вимикає режим редагування.
4. Показує повідомлення `Time saved`.

### `change_set_time(struct context *ctx, int direction)`

Змінює годину або хвилину.

`direction` може бути:

- `1` для UP;
- `-1` для DOWN.

Якщо редагується година:

- після 23 переходить на 0;
- перед 0 переходить на 23.

Якщо редагуються хвилини:

- після 59 переходять на 0;
- перед 0 переходять на 59.

### `start_set_date(struct context *ctx)`

Починає редагування дати.

Бере поточну дату з `now()` і записує в тимчасові поля:

- `set_day`;
- `set_month`;
- `set_year`.

Також перевіряє, щоб дата була в нормальних межах.

### `save_set_date(struct context *ctx)`

Зберігає дату.

Алгоритм:

1. Бере поточний час через `now()`.
2. Викликає `set_datetime()`, передаючи нову дату і старий час.
3. Показує `Date saved`.

### `change_set_date(struct context *ctx, int direction)`

Змінює день, місяць або рік.

Поле залежить від `ctx->date_field`:

- `0` - день;
- `1` - місяць;
- `2` - рік.

При зміні місяця або року функція перевіряє, чи не став день невалідним. Наприклад, якщо було 31 число, а користувач перейшов на квітень, день коригується до 30.

### `start_set_alarm(struct context *ctx)`

Починає редагування будильника.

Скидає поле редагування на перше поле - година.

### `save_set_alarm(struct context *ctx)`

Зберігає будильник в EEPROM через `save_alarm_settings()`.

Показує повідомлення:

```txt
Alarm saved
```

### `change_set_alarm(struct context *ctx, int direction)`

Змінює:

- годину будильника;
- хвилину будильника;
- стан ON/OFF.

Поле залежить від `ctx->alarm_field`:

- `0` - година;
- `1` - хвилина;
- `2` - ON/OFF.

### `start_alarm_ring(struct context *ctx)`

Запускає стан активного будильника.

Вона:

- ставить `alarm_ringing = true`;
- перемикає екран на `ALARM_SCR`;
- очищає LCD.

Сам звук вмикається не тут, а у `update_buzzer_and_led()`.

### `stop_alarm_ring(struct context *ctx)`

Зупиняє сигнал будильника.

Вона:

- ставить `alarm_ringing = false`;
- викликає `noTone(BUZZER_PIN)`;
- повертає екран на `CLOCK_SCR`;
- очищає LCD.

### `snooze_alarm(struct context *ctx)`

Відкладає будильник на 5 хвилин.

Алгоритм:

1. Бере поточний час.
2. Додає 5 хвилин.
3. Записує новий час у `snooze_hour` і `snooze_minute`.
4. Вмикає `snooze_active`.
5. Вимикає поточний сигнал.
6. Повертає екран годинника.

На екрані годинника при активному snooze показується `SNZ`.

### `check_alarm(struct context *ctx)`

Перевіряє, чи треба запускати будильник.

Алгоритм:

1. Якщо будильник уже дзвонить, нічого не робить.
2. Бере поточний час.
3. Якщо активний snooze і настав snooze час, запускає сигнал.
4. Якщо звичайний будильник вимкнений, виходить.
5. Якщо година і хвилина збігаються з часом будильника, запускає сигнал.

`last_alarm_key` потрібен, щоб будильник не перезапускався багато разів в одній і тій самій хвилині.

### `stopwatch_elapsed(struct context *ctx)`

Рахує, скільки часу пройшло в секундомірі.

Якщо секундомір на паузі, повертає накопичений час.

Якщо секундомір працює, повертає:

```txt
накопичений час + час з останнього старту
```

### `toggle_stopwatch(struct context *ctx)`

Перемикає секундомір:

- якщо працював - ставить на паузу;
- якщо був на паузі - запускає.

### `reset_stopwatch(struct context *ctx)`

Скидає секундомір у 0.

### `start_timer(struct context *ctx)`

Запускає countdown timer.

Обчислює тривалість:

```cpp
duration = (minutes * 60 + seconds) * 1000
```

Потім зберігає момент завершення:

```cpp
timer_ends_at = millis() + timer_remaining_ms;
```

### `pause_timer(struct context *ctx)`

Ставить таймер на паузу.

Рахує, скільки мілісекунд залишилось, і зберігає це в `timer_remaining_ms`.

### `resume_timer(struct context *ctx)`

Продовжує таймер після паузи.

Встановлює новий `timer_ends_at`.

### `reset_timer(struct context *ctx)`

Скидає таймер у режим налаштування.

Також вимикає buzzer через `noTone()`.

### `change_timer_value(struct context *ctx, int direction)`

Змінює хвилини або секунди таймера.

Поле залежить від `timer_field`:

- `0` - хвилини;
- `1` - секунди.

Хвилини обмежені від 0 до 99, секунди від 0 до 59.

### `update_timer(struct context *ctx)`

Перевіряє, чи завершився таймер.

Якщо час вийшов:

- `timer_running = false`;
- `timer_done = true`;
- залишок часу 0;
- екран перемикається на `TIMER_SCR`;
- LCD очищається.

Після цього buzzer і LED обробляються у `update_buzzer_and_led()`.

### `factory_reset(struct context *ctx)`

Скидає налаштування:

- будильник на `07:00 OFF`;
- snooze вимикається;
- збережений час у EEPROM скидається через `reset_saved_datetime()`;
- секундомір скидається;
- таймер ставиться на `01:00`;
- показується `Factory reset / Done`.

### `handle_buttons(struct context *ctx)`

Це головна функція обробки кнопок.

Вона:

1. читає всі кнопки через `read_pressed()`;
2. якщо дзвонить будильник, обробляє snooze або вимкнення;
3. якщо завершився таймер, будь-яка кнопка зупиняє сигнал;
4. якщо натиснуто MODE, перемикає екран;
5. якщо натиснуто SELECT, виконує дію залежно від екрана;
6. якщо натиснуто UP або DOWN, змінює значення залежно від екрана.

Це одна з центральних функцій у програмі.

### `update_buzzer_and_led(struct context *ctx)`

Керує buzzer і LED.

Якщо дзвонить будильник або завершився таймер:

- buzzer вмикається і вимикається кожні 250 мс;
- LED блимає разом із buzzer.

Якщо сигналу немає:

- buzzer вимкнений;

### `show_env(struct context *ctx)`

### `run_screen(struct context *ctx, enum screen active_screen)`

Це головний цикл одного screen.

Він виконує:

1. встановлює активний екран;
2. обробляє кнопки;
3. читає датчик;
4. оновлює таймер;
5. перевіряє будильник;
6. оновлює buzzer і LED;
7. показує поточний екран;
8. робить маленьку затримку 40 мс;
9. повертає поточний екран.

Ця функція дозволяє не дублювати однакову логіку в кожному screen.

### `clock_screen(...)`, `show_date_screen(...)`, інші screen-функції

Кожна з цих функцій просто викликає `run_screen()` зі своїм екраном.

Наприклад:

```cpp
enum screen clock_screen(struct context *ctx) {
    return run_screen(ctx, CLOCK_SCR);
}
```

Це зроблено для відповідності структурі skeleton проєкту: кожен screen має свою функцію.

## 17. Як працює весь цикл програми

1. `main()` запускає `INIT_SCR`.
2. `init_screen()` ініціалізує Arduino, LCD, RTC, DHT11, кнопки, buzzer, LED і EEPROM налаштування.
3. Після ініціалізації програма переходить у `CLOCK_SCR`.
4. Кожна ітерація викликає відповідну screen-функцію.
5. Screen-функція викликає `run_screen()`.
6. `run_screen()` обробляє кнопки, датчики, будильник, таймер, LED, buzzer і LCD.
7. Якщо користувач натискає MODE, екран змінюється.
8. Якщо час будильника збігається з поточним часом, програма переходить у `ALARM_SCR`.

## 18. Як працює налаштування часу

Послідовність:

1. Користувач переходить на `Set Time`.
2. Натискає `SELECT`.
3. Починається редагування години.
4. `UP/DOWN` змінюють годину.
5. `SELECT` переходить до хвилин.
6. `UP/DOWN` змінюють хвилини.
7. `SELECT` зберігає час.

Збереження виконується через:

```cpp
set_datetime(...)
```

Там час записується в програмний годинник, EEPROM snapshot і RTC.

## 19. Як працює налаштування будильника

Послідовність:

1. Користувач переходить на `Set Alarm`.
2. `SELECT` - редагування години.
3. `UP/DOWN` - зміна години.
4. `SELECT` - редагування хвилин.
5. `UP/DOWN` - зміна хвилин.
6. `SELECT` - ON/OFF.
7. `UP/DOWN` - перемикання ON/OFF.
8. `SELECT` - збереження.

Дані будильника записуються в EEPROM:

- година;
- хвилина;
- стан ON/OFF.

## 20. Як працює будильник

Функція `check_alarm()` постійно перевіряє поточний час.

Якщо:

```txt
current.hours == alarm_hour
current.minutes == alarm_minute
alarm_enabled == true
```

то викликається `start_alarm_ring()`.

Після цього:

- екран стає `ALARM_SCR`;
- `update_buzzer_and_led()` починає вмикати buzzer;
- LED блимає.

## 21. Як працює Snooze

Коли будильник дзвонить, натискання `SELECT` викликає `snooze_alarm()`.

Функція додає 5 хвилин до поточного часу і записує це в:

```cpp
snooze_hour
snooze_minute
```

Потім сигнал вимикається.

Коли настає snooze час, `check_alarm()` знову запускає будильник.

## 22. Як працює секундомір

Секундомір не використовує RTC. Він працює через `millis()`.

При старті запам'ятовується:

```cpp
stopwatch_started_at = millis();
```

При паузі накопичується пройдений час.

При reset усе скидається в нуль.

## 23. Як працює таймер

Таймер теж працює через `millis()`.

Коли користувач запускає таймер:

```cpp
timer_ends_at = millis() + timer_remaining_ms;
```

Потім `update_timer()` перевіряє, чи поточний `millis()` вже дійшов до `timer_ends_at`.

Якщо так:

- таймер завершується;
- `timer_done = true`;
- buzzer починає пищати.

## 24. Що казати, якщо питають про EEPROM

У проєкті EEPROM використовується для двох речей:

1. Збереження налаштувань будильника.
2. Збереження останнього встановленого часу і дати.

Для EEPROM використовується `EEPROM.update()`, щоб зменшити кількість зайвих записів.

Magic byte використовується для перевірки, чи в EEPROM уже лежать коректні дані.

## 25. Що казати, якщо питають про RTC

У проєкті використовується DS1302 RTC. Він підключений через `DAT`, `CLK`, `RST`, а не через I2C.

RTC ініціалізується у `clock_init()`.

При встановленні часу функція `set_datetime()` записує час у RTC через:

```cpp
rtc.SetDateTime(updated);
```

Також є перевірка валідності дати і часу, щоб уникнути невалідних значень.

## 26. Що казати, якщо питають про state machine

Програма побудована як state machine.

Список станів описаний у `screens.h`.

`main.cpp` містить нескінченний цикл і `switch`, який викликає функцію активного стану.

Кожна screen-функція повертає наступний стан.

Перевага такого підходу:

- легше додавати нові екрани;
- легше читати код;
- логіка кожного режиму відділена;
- немає одного величезного `loop()` з усім кодом.

## 27. Що казати, якщо питають про додаткові функції

У проєкті є такі додаткові функції:

### Snooze

Відкладає будильник на 5 хвилин.

### Stopwatch

Секундомір зі стартом, паузою і reset.

### Countdown timer

Таймер зворотного відліку. Коли доходить до нуля, вмикається buzzer.

### Factory reset

Скидає будильник, таймер, секундомір і збережений час.

Якщо треба назвати саме три додаткові функції, можна назвати:

1. Snooze.
2. Stopwatch.
3. Countdown timer.

Factory reset можна згадати як додаткову допоміжну функцію.

## 28. Потенційні питання на захисті

### Чому кнопки використовують `INPUT_PULLUP`?

Щоб не використовувати зовнішні pull-down резистори. Arduino має внутрішній pull-up резистор. Тому не натиснута кнопка читається як `HIGH`, а натиснута як `LOW`.

### Навіщо потрібен debounce?

Механічна кнопка при натисканні може кілька разів швидко перемикати сигнал. Debounce робить так, що одне натискання рахується один раз.

### Навіщо потрібен `context`?

Щоб зберігати стан програми в одному місці і передавати його між screen-функціями.

### Чому `main.cpp` такий короткий?

Бо вся логіка винесена в окремі модулі. `main.cpp` тільки керує state machine.

### Чому LCD wrapper потрібен?

Щоб код екранів не залежав напряму від бібліотеки `LiquidCrystal_I2C`. Якщо змінити LCD бібліотеку, треба буде змінювати тільки wrapper.

### Чому DHT11 читається не кожен цикл?

DHT11 не можна читати дуже часто. Тому читання виконується приблизно раз на 2 секунди.

### Як будильник не запускається багато разів за одну хвилину?

Для цього є `last_alarm_key`. Він запам'ятовує день і хвилину останнього запуску будильника.

### Як працює LED?

Якщо будильник увімкнений, LED світиться. Якщо будильник або таймер дзвонить, LED блимає разом із buzzer.

### Чому використовується `millis()`, а не `delay()` для секундоміра і таймера?

Тому що `millis()` дозволяє рахувати час без блокування всієї програми. Програма продовжує читати кнопки і оновлювати екран.

## 29. Чеклист для демонстрації

Перед захистом перевір:

- LCD показує стартове повідомлення;
- `MODE` перемикає всі екрани;
- `Time` показує час;
- `Date` показує дату;
- `Temp/Humidity` показує DHT11;
- `Set Time` змінює години і хвилини;
- `Set Date` змінює день, місяць, рік;
- після вимкнення/увімкнення відновлюється останній встановлений час;
- `Set Alarm` дозволяє виставити будильник;
- при `Alarm ON` світиться LED;
- будильник дзвонить у заданий час;
- `SELECT` під час дзвінка вмикає snooze;
- `MODE` під час дзвінка вимикає сигнал;
- stopwatch стартує, ставиться на паузу і скидається;
- timer налаштовується, стартує, ставиться на паузу і дзвонить після завершення;
- factory reset скидає налаштування.

## 30. Коротка відповідь, якщо треба пояснити весь проєкт за 1 хвилину

Це будильник на Arduino UNO. Він має LCD дисплей, RTC DS1302, DHT11, кнопки, buzzer і LED. Програма побудована як state machine: кожен режим є окремим екраном. `main.cpp` тільки перемикає екрани, а логіка винесена в `screens/app.cpp`. Стан програми зберігається в структурі `context`. Час і дата обробляються через `rtc_wrapper`, LCD через `lcd_wrapper`, датчик через `sensors`. Будильник можна налаштувати кнопками, його налаштування зберігаються в EEPROM, а при спрацюванні вмикається buzzer і блимає LED. Додатково реалізовані snooze, stopwatch і countdown timer.
